### Operational Baseline (Strengths)

Certain architectural decisions in this proposal are structurally sound and require no modification. 

* [cite_start]**BYOM (Bring Your Own Model) Architecture:** Forcing users to supply their own LLM keys (OpenAI, Anthropic) [cite: 53, 315] is the only financially viable path for an NGO/academic project. It eliminates recurring API overhead for CDLI and avoids rate-limiting bottlenecks at the centralized server level.
* [cite_start]**Dual Transport Implementation:** Supporting both STDIO for local clients (Cursor/Claude) and HTTP/SSE for the web widget [cite: 47, 48, 229, 230] correctly serves both target demographics (casual web researchers and hardcore desktop tool users) without maintaining two separate codebases.
* [cite_start]**Tool Schema Definition:** The explicit adherence to the official MCP TypeScript SDK formatting and JSON Schema [cite: 139-152] ensures compatibility. You have correctly parameterized inputs for tools like `advanced_search` and `cqp_query` rather than relying on unstructured text parsing.
* [cite_start]**Mitigation Awareness:** Implementing an AbortController with an 8-second timeout for slow CDLI APIs [cite: 321] [cite_start]and hard-limiting the LLM to 5 consecutive tool calls [cite: 321] demonstrates a basic understanding of LLM loop failures.

### Architecture & Technical Feasibility Flaws

* **The `/paper` Command is a Fantasy:** The mentor requirements explicitly call for a "Research Paper Generation Agent." [cite_start]You included a `/paper` slash command [cite: 279] but provided zero architectural detail on how this operates. Drafting a comprehensive academic paper requires multi-step reasoning, massive context windows (ingesting full ATF transliterations for multiple artifacts), and cross-referencing. Doing this inside a browser-based React widget hitting a user's API key is highly unstable. If the user requests a paper on 50 tablets, the context window will explode, or the browser tab will freeze processing the streamed markdown. You need a dedicated agentic workflow (e.g., LangGraph) for this, not a simple chat completion loop.
* [cite_start]**CQP-to-RDF Endpoint Volatility:** You plan to expose the CQP-to-RDF API via the `cqp_query` tool [cite: 195-205]. The mentor description explicitly notes this is an *experimental* system. Graph database queries are notoriously slow and resource-heavy. [cite_start]Your 8-second timeout [cite: 321] will likely kill valid, complex linguistic queries before they return. You have no strategy for handling asynchronous or long-polling requests for heavy graph queries.
* [cite_start]**"Trimming" JSON is Fatal:** Your mitigation for overly long tool responses is that "Tool output is trimmed before being fed back to the LLM"[cite: 321]. You cannot blindly trim a JSON response; doing so invalidates the JSON structure, causing the LLM parser to fail and hallucinate. You must implement structural pagination at the API layer, or return a dedicated subset of keys (e.g., returning only IDs and titles, forcing the LLM to call `get_artifact` for full text). 

### Scalability Risks

* [cite_start]**In-Memory Caching is Inadequate:** You propose caching frequent queries "in memory for ~30 seconds"[cite: 321]. This is useless for an LLM system. LLMs rarely generate the exact same string twice (e.g., `query: "rain rituals"` vs `query: "rituals regarding rain"`). You will experience a massive cache miss rate. You need a semantic cache (like Redis with vector embeddings) or, at the very least, parameter-level normalization before caching. [cite_start]Furthermore, holding cache in-memory on a Node.js server container [cite: 334] destroys statelessness, making horizontal scaling impossible.
* [cite_start]**SSE Connection Saturation:** You plan to use Server-Sent Events (SSE) for the web widget[cite: 314]. Node.js handles concurrent SSE connections poorly without aggressive tuning because each connection holds the socket open. [cite_start]If a university class of 100 students logs on to use the tool simultaneously, your single Docker container [cite: 332] will run out of file descriptors or max out event loop ticks. 

### Logic Gaps & Unvalidated Assumptions

* [cite_start]**The "One-Line" CakePHP Integration:** You claim integration requires only adding a `<div>` and a `<script>` tag[cite: 311, 312]. This ignores the reality of integrating modern React bundles into legacy monolithic PHP systems. You have entirely ignored Content Security Policy (CSP) headers blocking your external SSE connections, CSS global scope polluting your widget styling, and cross-origin resource sharing (CORS) configurations for `mcp.cdli.earth`. 
* [cite_start]**Widget Development Timeline:** You have allocated just 3 weeks (Weeks 7-10) to build a production-grade React widget [cite: 369-372] [cite_start]with markdown streaming, secure API key storage, complex slash-command workflows, and persistent chat history[cite: 286]. [cite_start]Building a resilient UI that correctly renders arbitrary MCP tool-call states and handles SSE reconnection logic [cite: 321] is a massive undertaking. This timeline is overly optimistic for a solo developer.

### Premortem

**Date of Failure:** November 2026.
**Cause of Death:** Context-Window Exhaustion and API Throttling.

Within weeks of launch, users begin treating the widget like ChatGPT, bypassing the basic search tools and immediately triggering the `/paper` command to synthesize 30+ artifacts at once. [cite_start]The MCP server obliges, pulling hundreds of kilobytes of JSON metadata and ATF transliterations[cite: 166], and shoves it back to the client. 

Because the system relies on the user's BYOM API key running client-side, the user's browser attempts to send a 150,000-token prompt to OpenAI/Anthropic. The API instantly rejects the request due to token limits or tier restrictions. The user gets a generic "API Error" in the widget. Meanwhile, the few requests that do go through cause the LLM to get stuck in a loop, rapidly firing paginated requests at the CDLI database to gather more data, triggering CDLI's internal rate limits and bringing the core `cdli.earth` REST APIs down for everyone. 

The project fails because it relies on the LLM to manage its own context size, assumes client-side API limits will mirror enterprise limits, and fails to sandbox the ingestion of raw cuneiform text data.
