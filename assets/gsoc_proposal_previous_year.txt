Abstract 
 
This proposal outlines the development of an AI Agent for Apache HertzBeat using the Model Context Protocol (MCP), enabling natural language interaction with monitoring capabilities. The project will implement an MCP-compliant server and client architecture integrated with SpringAI, along with a chat-based frontend for the HertzBeat dashboard. This enhancement will make HertzBeat's monitoring features more accessible while serving as a reference implementation for AI integration in monitoring tools.

3    Background 
 
Apache HertzBeat provides comprehensive monitoring capabilities, alerts and connects with a variety of data sources. But it lacks the intuitive natural language interaction. With the rise in AI and LLMs, and those being integrated with the most basic to complex applications, having an AI agent that can execute all the features becomes the need of the hour.  
Users must navigate traditional dashboards and forms to query monitoring data or configure new targets—a process that can be time-consuming and requires technical familiarity with the platform. This problem can be solved effectively by implementing a conversational agent, which will surely lead to wider adoption of HertzBeat.  All functions/APIs are already in place to set up monitoring targets, querying information, trigger alerts etc. We need to effectively curate on how to leverage these functions as tools for the LLM/AI agent to execute them based on the user's demands. Using MCP as a communication protocol is a really good approach. 
Implementing an MCP server also ensures that the users can access the data from third-party apps like Cursor, WindSurf, Claude Desktop, etc. Moreover, the support for MCP servers is ever increasing.

4.2.3 MCP Client and AI Agent Service 
 
1.	Make an MCP client service class that connects with the server and accesses all the available tools. 
2.	Expose the method that can be used by the Chat Controller to connect to this client and pass in the user queries 
3.	Add history to the user query for additional context using the Conversation Service 
4.	Add relevant system prompts. This also helps in optionally structuring the data. For example, if we want metrics data to always return in the form of a chart, we can mention that. Or conditionally append the same to the prompt if certain tools are decided to be called by LLM. 
5.	Add the tool calling and execution logic via the Agent Service, in integration with the chosen LLM provider i.e executing the tool and giving the response back to LLM. 6. LLM knows about all the tools, and can decide which tools to call in what order according to the parameters and user query. 
7. If LLM feels that it lacks certain information based on the parameters of the tools, it can revert back to ask the user for more and specific information. Or link them to read the user guide for obtaining keys etc.

8    Results for the Apache community
